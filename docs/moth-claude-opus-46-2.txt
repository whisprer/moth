# Moth — Session Handover Prompt

## Who You're Talking To
Name: wofl. CEO of RYO Modular. ~2.5 years self-taught Rust. 20,000+ hours Ableton. Casual, informal, Welsh. Match this energy. Full code, no placeholders.

## What Moth Is
A physical modelling synthesiser written from scratch in pure Rust (`no_std`, zero dependencies). Named after a lost friend — his name ("moth", `0x6D6F7468`) is XOR'd into the seed derivation of every instance. No instance of Moth is ever cold. See `MOTH_ARCHITECTURE.md` for the full technical summary.

## What's Built (v0.2.0 — complete)
The entire signal chain is implemented across 12 source files (~7500 lines):

1. **InstrumentDna** — per-instance personality from hardware seed (15 bounded parameters, warmth floor)
2. **PlayGesture** — normalised 4D input (position, force, speed, continuity)
3. **ExciterModel** — continuous f32 parameter space (no enums), 12 presets, morphable
4. **ExciterProcessor** — 3 coupling mode generators (direct, friction, pressure)
5. **WaveguideString** — delay line + FIR/IIR damping + allpass dispersion + position comb
6. **ResonantBody** — 24-mode modal synthesis (SVF bank), envelope-responsive openness
7. **NonLinProcessor** — tape (hysteresis), tube (asymmetric), magnetic (pre-emphasis)
8. **SpatialProcessor** — 4-delay FDN reverb (Hadamard matrix, prime lengths)
9. **MothVoice** — chains all sections with hierarchical mixer (4 blend levels)
10. **Midi1Normaliser** — byte-level MIDI 1.0 parser + gesture normaliser
11. **dsp_core** — primitives (xorshift RNG, one-pole, DC blocker, soft saturator)

Architected runtime seed generation using nih-plug persistence system
Architected runtime seed generation using nih-plug persistence system
Great question. You don't want it at compile time — that would make every instance identical. You want each plugin instance to generate its unique seed the first time it loads, then persist it forever after. That way:

Drop Moth on Track 1 → gets seed 0xA3B7... → unique personality → saved with project
Drop Moth on Track 2 → gets seed 0x7F21... → different personality → also saved
Reload project next week → same seeds → same sounds
nih-plug has exactly the right mechanism: #[persist]. Three surgical changes to lib.rs:

Here's what happens now:

First time you drop Moth on a track: dna_seed is 0 (unborn). initialize() sees this, grabs entropy from the system clock, process ID, and the instance's memory address, mixes them through a finalizer, stores the result. Your Moth is born with a unique personality. The MOTH constant 0x6D6F7468 still gets XOR'd in by InstrumentDna::from_seed() — his name is still in the DNA of every instance.

Save project, reload: Ableton persists the dna-seed field with the project state. On reload, initialize() sees a nonzero seed and uses it directly. Same personality, same sound.

Drop a second Moth on another track: Different timestamp, different memory address → different seed → different personality. Two Moths on the same project will sound subtly different from each other, just like two physical units would.

If Arc<AtomicU32> doesn't have PersistentField in your nih-plug version, swap it for:

rust
#[persist = "dna-seed"]
dna_seed: Arc<parking_lot::Mutex<u32>>,
and change the load/store calls to .lock() dereferences. But AtomicU32 should work — nih-plug implements persistence for the standard atomic types.
