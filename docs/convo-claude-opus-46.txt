Why continuous axes, not enums
The core insight is that physical excitation mechanisms exist on a continuum, not in discrete categories. A bow isn't a fundamentally different thing from a pluck — it's friction-dominated continuous energy transfer vs impulse-dominated direct injection. By parameterising along energy_continuity and the three coupling weights, you get smooth morphing between any two excitation characters for free. Named presets become bookmarks in a continuous space rather than ontological categories. This directly serves the morphing pillar — if you had enum ExciterType { Bow, Pluck, Breath }, you'd need bespoke crossfade logic for every pair, and you'd still never represent the spaces between them. The f32 axes make the interpolation path itself musically meaningful.

Understood and agreed. Now here's what I'd challenge:

Challenges
1. The coupling sum-to-1.0 constraint is too restrictive.

This is the big one. If coupling_direct + coupling_friction + coupling_pressure = 1.0, you've created a mixing triangle — boosting one axis necessarily attenuates the others. But real hybrid excitations layer mechanisms independently. Col legno tratto (bowing with the stick) is simultaneously high friction and significant direct contact energy. A didgeridoo involves pressure-driven oscillation with lip friction. Constraining the sum means you can never have full-strength friction plus meaningful direct injection — you're forced to trade off.

My recommendation: let all three be independent 0.0–1.0 ranges. Normalise at the output stage if you need to control total energy, not at the model definition. This also makes the morphing system simpler — you just lerp each axis independently instead of keeping them on a simplex.

2. Missing: spectral shape / exciter hardness.

A fingertip pluck, a plastic pick, and a harpsichord quill all have {continuity: 0.0, direct: 1.0} — but they sound radically different because the impulse bandwidth differs. The current model can't distinguish them. You likely need a hardness: f32 (or spectral_tilt) parameter — 0.0 = soft/warm (finger, felt mallet) → 1.0 = hard/bright (pick, metal beater). This shapes the initial noise burst spectrum for direct injection, the friction curve sharpness for bow, and the reed-opening attack profile for pressure. It's a fundamental timbral axis that's currently absent.

3. Stochasticity needs specification before implementation.

stochasticity: f32 as a concept is clear but where does the randomness apply? Options that produce very different results: amplitude modulation of exciter output, position jitter on the vibrator, timing jitter for multiplicity events, spectral noise injection. For granular/rain (stochastic=0.95, mult=32), you probably want timing + position jitter across the multiple contact points. For a breathy flute (stochastic=0.15, pressure=1.0), you want turbulence noise mixed into the airflow. These are different noise injection topologies. I'd suggest stochasticity controls multiple noise injection points simultaneously, with the coupling mode determining which topology dominates.

4. Minor: PlayGesture.speed is semantically overloaded.

For a bow, speed is bow velocity (critical — it directly determines friction energy). For breath, it maps to airflow rate. For a pluck, it's arguably meaningless. The normalisation from heterogeneous inputs (pitch bend rate, MPE X delta, breath CC) into a single speed field is fine at the gesture level, but each coupling mode will need to interpret speed differently in its physics. Worth documenting explicitly rather than discovering during implementation.

5. Non-issue (confirming): PlayGesture.continuity: bool vs ExciterModel.energy_continuity: f32 — these are distinct and correctly separated. The bool is player intent (gate held), the f32 is model character (impulse vs sustained injection). A held note with energy_continuity: 0.0 gives you a single impulse that rings out while the gate is open. Good.

What Mutable Instruments Elements actually does
The Seed → Signature pipeline
Line 120 of elements.cc:

cpp
part.Seed((uint32_t*)(0x1fff7a10), 3);
```

That address — `0x1fff7a10` — is the STM32F4's **hardware-fused unique device ID**. 96 bits. Baked into silicon at the fab. Every chip on earth has a different one. It's read *once* at boot.

Then `Part::Seed()` does the beautiful bit. It takes that unique silicon fingerprint and runs it through an LCG (linear congruential generator) with the constant `0xf0cacc1a` (focaccia — Émilie's humour), XORing and scrambling. Then it *peels off 3 bits at a time* — five extractions from the scrambled hash — to set:

1. **`resonator_modulation_frequency`** — how fast the internal position LFO runs (0.4–1.2× range)
2. **`resonator_modulation_offset`** — the stereo image shift of the modal resonator (0.05–0.15)
3. **`reverb_diffusion`** — how smeared the reverb is (0.55–0.70)
4. **`reverb_lp`** — reverb brightness (0.70–0.90)
5. **`exciter_signature`** — a 0.0–1.0 float passed directly into the exciter models

Five parameters. All derived deterministically from one unrepeatable seed. All within *carefully bounded sweet-spot ranges* — never the full 0.0–1.0, always a *musically vetted* sub-range where every value sounds good.

### How signature propagates through the signal chain

`exciter_signature` does two specific things in the exciter models:

**In `ProcessPlectrum` (pluck):** `signature_` scales the initial *negative* pre-pluck displacement — `impulse = -amplitude * (0.05f + signature_ * 0.2f)`. This means each unit's pluck has a slightly different ratio of initial pull-back to release snap. The pluck *character* varies per unit — how much the virtual plectrum bends before releasing.

**In `ProcessGranularSamplePlayer` (blow/breath noise source):** `signature_ * 8192.0f` offsets into the noise wavetable. Each unit starts reading from a different point in the noise buffer, so the turbulence texture of every unit's breath exciter is unique.

And then the resonator-side parameters (modulation frequency, modulation offset) mean that even with *identical* knob positions, two Elements modules produce subtly different stereo imaging and internal motion. The reverb parameters mean the spatial tail differs too.

### Why this is genius — the design principles

**1. Every parameter lives in a pre-vetted sweet spot band.** The modulation frequency range is `0.4–1.2 / kSampleRate` — not `0.0–10.0`. The reverb diffusion range is `0.55–0.70` — not `0.0–1.0`. Émilie chose these ranges because *every value within them sounds good*. The seed doesn't pick a random point in parameter space; it picks a random point in the *musically valid* subspace. This is why "all possible things you select are sweet spots."

**2. Everything affects everything but through constrained coupling.** The signature changes the pluck character which feeds into the resonator whose own modulation rate and stereo offset are also seed-derived. The reverb diffusion and LP are seed-derived too. So the *entire* signal chain has per-unit variation — but each point of variation is bounded tightly enough that the interactions never produce garbage. It's cross-modulation within safe envelopes.

**3. Determinism, not randomness.** Same chip → same signature → same character every power cycle. Your unit has a *personality*, not noise. It's the difference between a violin having its own voice vs just being out of tune.

**4. The user never sees or controls these parameters.** There's no "signature" knob on the panel. There's no menu to adjust reverb diffusion to the third decimal place. These are the instrument's *nature*, not its settings. The user shapes the instrument; the signature shapes the instrument *back*.
```

exciter_dsp.rs — The three coupling engines + processor
ExciterProcessor is the runtime engine. Instantiated per-voice with DNA + sample rate. The process() method fills an output buffer from model + gesture.

The three coupling mode engines:

Direct — On gate trigger, the impulse envelope jumps to force and decays exponentially. Decay rate is controlled by spectral tilt: soft (felt mallet) = slow decay/long contact, hard (metal beater) = fast decay/sharp click. A DNA-signature-derived pre-impulse creates a negative displacement before the strike (like Émilie's plectrum pull-back — 0.05 + signature * 0.20). In continuous mode (energy_continuity near 1.0), the envelope sustains at continuity × force — the e-bow behaviour. The signal itself is shaped white noise at the envelope level.

Friction — Directly inspired by Elements' ProcessFlow. A bistable particle alternates between +0.5 and -0.5 states with a probability controlled by speed^4. Between flips, noise proportional to velocity is added. This produces the characteristic stick-slip sawtooth-like energy of a bowed string. DNA coupling_curve_shape scales the transition threshold. stochastic_bias shifts the flip probability asymmetrically. Output scaled by force (bow pressure).

Pressure — Simplified reed/jet model inspired by Elements' tube.cc. Reed opening follows 0.8 - 0.2 * closure_rate * pressure — starts mostly open, closes under high pressure (overblow). Turbulence noise is low-passed and mixed into the flow, proportional to stochasticity × airflow. A tiny feedback term (-0.1 × prev_output) gives the exciter a self-oscillation tendency even before the vibrator waveguide provides its own feedback. DNA coupling_curve_shape modifies reed stiffness.

Mixing: the three signals are weighted by their respective coupling strengths, normalised by total_coupling when it exceeds 1.0 (prevents energy spikes from hybrid excitations), then passed through the spectral tilt filter (one-pole LP, cutoff derived from tilt × DNA.tilt_bias), soft-saturated, and DC-blocked.

Gesture smoothing: force and speed are linearly interpolated per-sample across the block, matching Elements' strength_increment anti-zipper pattern.

The vibrator: what it does
WaveguideString is a digital waveguide delay line — the canonical physical model of a vibrating string. Exciter output goes in, pitched audio comes out.

The signal path: excitation is added to the delay line at the write head. Each round trip through the delay, the signal passes through a two-stage damping chain (FIR averaging filter for frequency-dependent decay + IIR one-pole lowpass for brightness shaping), scaled by a feedback gain (energy loss per round trip), and optionally through an allpass filter for dispersion (inharmonicity — piano stiffness, metallic bell character). The output is read from a comb-filtered tap point whose offset is controlled by position, using Elements' clamping formula (0.5 - 0.98 * |pos - 0.5|) to avoid degenerate comb settings.

Pitch is set by delay line length: sample_rate / frequency. Changes are smoothed per-sample across the block to prevent clicks on pitch bends. DNA delay_micro_offset adds a fractional-sample offset — each Moth's intonation is subtly unique, like bridge placement tolerance.

Damping maps quadratically from 0 (rapid decay) to 1 (near-infinite sustain). Brightness controls how fast upper partials decay relative to the fundamental. Dispersion drives the allpass with DNA dispersion_asymmetry making each instance's inharmonicity character slightly different.

The delay line is 4096 samples (16KB per voice) — supports fundamentals down to ~11.7 Hz at 48kHz.

The resonant body: what it does and why
24 parallel Chamberlin SVF bandpass filters, each one a resonant mode of the virtual body. The vibrator's output excites all modes simultaneously; each mode rings at its own frequency, Q, and amplitude. The sum is the body's voice.

Geometry drives mode spacing via a stiffness parameter (directly from Elements' approach, from Penttinen's morphing work): geometry=0.0 compresses modes together (tube/air column), 0.25 gives harmonic spacing (ideal string resonance), 0.5 spreads them slightly (wooden box), 1.0 spreads them wide (bell/shell).
